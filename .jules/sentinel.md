## 2024-03-28 - SSRF in Data Source Configurations and Proxies
**Vulnerability:** Server-Side Request Forgery (SSRF) exists in `/api/data-sources` (POST/PUT) and `/api/introspect` (POST) because user-provided URLs (`apiUrl` and `url`) are not validated before being saved to the database or used in backend HTTP requests via `axios`. This allows users to set arbitrary URLs (e.g., `file://`, `ftp://`, internal network IPs) that the backend will try to access, potentially exposing internal services or local files.
**Learning:** The application acts as a proxy/gateway to external APIs based on user configuration, making it a high-risk vector for SSRF. The root cause is a lack of strict protocol and hostname validation on user-submitted URLs.
**Prevention:** Implement strict URL validation for all user-provided endpoints. Ensure only `http:` and `https:` protocols are permitted before saving configurations or initiating outgoing backend requests.

## 2024-05-02 - SSRF in Data Proxy Endpoints (Defense-in-depth)
**Vulnerability:** Even if the database configurations only contain valid `http:`/`https:` URLs, proxy endpoints such as `/api/data/:entity` dynamically build strings for target URLs without re-validating the final URLs before calling external resources using `axios` or `graphql-request`. This could still leave a surface for SSRF if URL concatenation logic produces malformed or dangerous protocols.
**Learning:** Validation at the data entry point (database saving) is not always sufficient, especially in complex applications where URLs may be altered or appended to before being used. A defense-in-depth approach is required where outgoing network requests validate the URL right before the request is executed.
**Prevention:** Always use strict URL parsing (e.g., `new URL(url).protocol`) immediately prior to any backend network call, regardless of where the data originated, returning a 400 Bad Request on invalid protocols.
