# Tyk API Gateway: Security & Observability

This document details the architecture, best practices, and integration strategies for securing and monitoring APIs using Tyk Operator within this application stack.

---

## 1. Security Architecture

### 1.1 OAuth2 Proxy Integration
Your infrastructure uses `oauth2-proxy` combined with Tyk to secure endpoints. The typical flow is:
1. **Ingress/Client:** Request arrives at the ingress controller.
2. **OAuth2 Proxy:** Validates the user session via an OpenID Connect (OIDC) provider.
   - If not authenticated: Redirects to login.
   - If authenticated: Injects the identity token (e.g., JWT) into the `Authorization: Bearer <token>` header and forwards to the gateway.
3. **Tyk Gateway:**
   - Receives the request with the JWT.
   - Validates the JWT Signature (against JWKS endpoint).
   - Validates JWT claims (`exp`, `iss`, `aud`).
   - Applies the appropriate `SecurityPolicy` based on the token.

### 1.2 JWT Validation in Tyk
In `k8s/tyk/backend-api.yaml`, we've enabled JWT authentication. Ensure your Tyk installation is configured with the correct OIDC provider's JWKS (JSON Web Key Set) URL so Tyk can verify the signature.

```yaml
# Inside the Tyk ApiDefinition
use_jwt: true
jwt_source: "Authorization"
# Requires adding the JWKS URL in the Tyk Dashboard or Global config
```

### 1.3 Rate Limiting and Quotas
Rate limiting prevents abuse and ensures service availability (DDoS mitigation).
- The `SecurityPolicy` (`k8s/tyk/security-policy.yaml`) sets a baseline of **100 requests per minute**.
- A quota of **10,000 requests per hour** prevents excessive long-term usage.
- Adjust these values per-API or per-client-tier in your `SecurityPolicy`.

### 1.4 Mutual TLS (mTLS) [Recommendation]
For Zero-Trust network architecture, it is recommended to enable mTLS between Tyk and the upstream services (the Backend and Frontend pods). This ensures that only Tyk can communicate directly with your backend services.

### 1.5 Backend Security (SSRF Prevention)
As the backend proxies requests dynamically (`/api/data/:entity`), it is crucial that Tyk strips or strictly sanitizes paths before forwarding.
Tyk handles basic path routing, but the backend must enforce strict URL protocol validation (`http:` / `https:` only) and prevent path traversal attacks when making external calls based on user input.

### 1.6 Microfrontend Integration Security
To control which host applications can integrate and load your microfrontends (e.g., via Module Federation), the primary and most effective browser-enforced mechanism is **Cross-Origin Resource Sharing (CORS)**.

By configuring CORS strictly on the `frontend-api` Tyk definition (or the upstream web server), you can ensure that browsers will only execute the microfrontend code if it's loaded from a trusted domain.
```yaml
# Inside the Tyk ApiDefinition for the frontend
cors:
  enable: true
  allowed_origins:
    - "https://trusted-host-app1.example.com"
    - "https://trusted-host-app2.example.com"
  allowed_methods:
    - "GET"
    - "OPTIONS"
```
Additionally, if you need stronger guarantees than CORS (which can be bypassed by non-browser clients), consider serving the microfrontend manifest (`remoteEntry.js` or equivalent) behind an authentication layer that requires a service-to-service token exchange or a specific valid user session.

## 2. Observability Architecture

### 2.1 Tyk Pump
Tyk Pump is the central data processor that takes analytics generated by the Tyk Gateway nodes and forwards them to various backend sinks (e.g., Prometheus, Splunk).

### 2.2 Prometheus & Grafana Integration
To monitor API performance (Latency, Error Rates, Request Volumes):
1. **Enable Prometheus Pump:** Configure Tyk Pump to expose a Prometheus scraping endpoint.
   ```yaml
   # Tyk Pump Config (pump.conf or Helm Values)
   pumps:
     prometheus:
       type: "prometheus"
       meta:
         listen_address: ":9090"
         path: "/metrics"
   ```
2. **Prometheus Scraping:** Configure your Prometheus instance to scrape `http://tyk-pump:9090/metrics`.
3. **Grafana Dashboards:** Import the official Tyk Grafana dashboards (available on Grafana Labs) to visualize API usage, response times, and HTTP status codes.

### 2.3 Splunk Integration
To retain long-term audit logs and detailed request/response payloads:
1. **Enable Splunk Pump (or Syslog):** You can configure Tyk Pump to send logs directly to Splunk using the Splunk HTTP Event Collector (HEC).
   ```yaml
   # Tyk Pump Config (pump.conf or Helm Values)
   pumps:
     splunk:
       type: "splunk"
       meta:
         collector_url: "https://http-inputs-splunk.example.com/services/collector/raw"
         token: "YOUR_HEC_TOKEN"
   ```
2. **Data Included:** Ensure you configure Tyk to log what is necessary (e.g., headers, path, status) without logging sensitive PII or raw passwords in the payloads.

### 2.4 Distributed Tracing (OpenTelemetry / Jaeger) [Recommendation]
To trace requests as they flow from the Ingress -> Tyk -> Backend -> Database, enable tracing in Tyk. Tyk supports OpenTelemetry. This will allow you to see the exact breakdown of latency across microservices in Grafana Tempo or Splunk APM.
