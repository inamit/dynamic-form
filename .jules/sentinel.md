## 2026-03-26 - [Path Traversal in Dynamic Proxy]
**Vulnerability:** The backend dynamically proxies requests to external endpoints via `/api/data/:entity/:id` and `/api/enums/:enumName` by simply appending user input to a configured URL base.
**Learning:** These endpoints were vulnerable to path traversal because `id` and `enumName` inputs were unchecked and directly included in `axios.get(url + '/' + input)`. An attacker could pass `../` to access other backend proxy targets unintentionally.
**Prevention:** Always validate and sanitize user-provided values used to construct paths or URLs in proxies, explicitly blocking traversal patterns like `/` or `..`.

## 2025-02-28 - [Critical] SSRF Vulnerability in Proxy Endpoints
**Vulnerability:** The application dynamically proxies requests to external endpoints without validating the target URL. Endpoints such as `/api/introspect`, `/api/data-sources`, `/api/schemas`, and others accept user-provided URLs or retrieve URLs from the database and pass them directly to `axios` or `graphql-request`. This allows an attacker to make the server send arbitrary HTTP requests to internal networks or unauthenticated internal services, resulting in a Server-Side Request Forgery (SSRF) vulnerability.
**Learning:** External or dynamically-provided URLs must always be validated to ensure they adhere to expected protocols (e.g., `http:` and `https:`) and domains to prevent the server from being used as a proxy to attack internal infrastructure or access restricted resources (like `file://` or `gopher://`).
**Prevention:** Implement a strict URL validation mechanism using the built-in Node.js `URL` class. Ensure that the protocol is exactly `http:` or `https:` before passing any URL to an HTTP client like `axios`. In addition to protocol validation, consider restricting requests to a whitelist of allowed domains or IP ranges if the use case permits.
