## 2024-03-28 - SSRF in Data Source Configurations and Proxies
**Vulnerability:** Server-Side Request Forgery (SSRF) exists in `/api/data-sources` (POST/PUT) and `/api/introspect` (POST) because user-provided URLs (`apiUrl` and `url`) are not validated before being saved to the database or used in backend HTTP requests via `axios`. This allows users to set arbitrary URLs (e.g., `file://`, `ftp://`, internal network IPs) that the backend will try to access, potentially exposing internal services or local files.
**Learning:** The application acts as a proxy/gateway to external APIs based on user configuration, making it a high-risk vector for SSRF. The root cause is a lack of strict protocol and hostname validation on user-submitted URLs.
**Prevention:** Implement strict URL validation for all user-provided endpoints. Ensure only `http:` and `https:` protocols are permitted before saving configurations or initiating outgoing backend requests.

## 2024-04-05 - [SSRF Protocol Smuggling Fix]
**Vulnerability:** The backend `apps/backend/src/index.ts` was proxying external data sources dynamically retrieved from the DB (e.g. `ds.apiUrl`) via `axios` and `graphql-request` without strictly enforcing URL protocols.
**Learning:** Even when reading trusted configuration properties from the database (like an internal API URL string), those URLs must still be rigorously validated. If a configuration is modified, or if local path parameters manipulate the structure, an SSRF via protocol smuggling (e.g., using `file://` or `gopher://` strings) becomes possible.
**Prevention:** Always validate and parse dynamic base URLs against explicitly allowed schemes (e.g., `http:` and `https:`) using Node's `new URL()` constructor before making an outbound request.

## 2024-05-18 - [SSRF URL Normalization Bypass]
**Vulnerability:** The application was validating user-provided URL strings against a protocol whitelist (`http:`, `https:`) using Node's `new URL()`, but passing the original, unmodified URL string to `axios.post`. This creates a vulnerability via URL parser differentials where `axios` might interpret malformed URLs differently than Node's `URL` parser.
**Learning:** Always use the parsed and normalized URL generated by the validation routine (e.g., `url.href`) when making the actual outgoing request. Returning and using the original string after validating it allows malicious payloads to bypass the check.
**Prevention:** Make validation functions return the sanitized and normalized primitive (like `url.href`), and enforce its usage at the HTTP client call sites.

## 2024-05-20 - [Path Traversal bypass via URL encoding]
**Vulnerability:** The application was vulnerable to path traversal because it validated route parameters (like `entityName`, `id`) against a deny-list (`param.includes('..')`). Express.js decodes parameters, but this simple string check could be bypassed using URL encoding (like `%2e%2e%2f` for `../`) if the string was later passed to `axios` or another URL constructor.
**Learning:** Deny-list validation for security concepts like path traversal is notoriously fragile and prone to bypasses. It assumes you can enumerate all possible bad inputs.
**Prevention:** Always use strict allow-list validation. For dynamic route parameters that should only contain standard characters (e.g., entity names or numeric/UUID IDs), enforce validation using a strict regex (e.g., `/^[a-zA-Z0-9_-]+$/`).

## 2024-04-12 - Fix Mass Assignment in DataSource creation
**Vulnerability:** The `DataSourceService` was passing user-provided JSON payloads directly to Prisma via `this.dataSourceRepo.createDataSource(data)` and `this.dataSourceRepo.updateDataSource(id, data)`. This created a mass assignment vulnerability where an attacker could inject arbitrary database fields or override relational mappings (e.g. `id` or other relational dependencies) in the `DataSource` table.
**Learning:** When dealing with ORMs like Prisma, simply passing `req.body` (or any unvalidated `data` object originating from a user request) straight into creation or update routines exposes all model attributes to manipulation.
**Prevention:** Always explicitly define a whitelist of permitted fields (for instance, via destructuring: `const { name, apiUrl, apiType, headers, endpointsQueries } = data;`) before passing the object into the ORM layer, effectively isolating the system from extraneous or malicious fields in incoming payloads.
